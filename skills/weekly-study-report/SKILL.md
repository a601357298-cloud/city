# Weekly Study Report

目标：把最近一周的学习笔记、飞书文档和相关记录，整理成一份简洁、正式、可提交的中文周报，并尽量输出 Word（.docx）。

## 工作流

1. **先找真实材料，不要空写**
   - 优先搜索最近 7 天的飞书文档 / wiki / 学习笔记。
   - 如果用户明确说“都在飞书文档中”，先从飞书文档抓内容，再动笔。
   - 如有用户当天/当周直接发来的聊天笔记，也纳入素材。

2. **抽主线，不抄流水账**
   - 识别本周 3-5 个主题模块，例如：
     - Python / 文件处理 / 正则
     - openpyxl / NumPy / Pandas
     - 国产数据库 / 备份恢复 / 权限管理
     - WPS / 图数据库 / 其他培训专题
   - 只保留对“学习收获 + 审计工作联系”有价值的内容。

3. **周报结构默认如下**
   - 标题：
     - 中铝集团审计人员数字化审计能力转型
     - 专项培训一周学习心得
   - 一级标题：
     - 一、感恩与启程
     - 二、学习内容回顾与心得体会
     - 三、未来学习计划
   - 二级标题（放在第二部分下）
     - （一）……
     - （二）……
     - （三）……

4. **写作规则**
   - 风格：正式、自然、像单位材料，但不要空洞套话堆满。
   - 必须结合本周真实学习内容，不要写成万能模板。
   - 学习内容不必过细，不逐条罗列语法点；要概括成能力提升与应用理解。
   - 尽量自然带出与审计工作的联系，如数据处理、风险识别、底稿整理、权限控制、异常筛查等。
   - 篇幅控制：默认 1-2 页 Word 为宜，不过长。

5. **生成 Word 的优先路径**
   - 优先使用本地已安装的 `docx` Node 包生成 `.docx`。
   - 全局 Node 模块可能默认不可见，先设置：
     - `export NODE_PATH="$(npm root -g)"`
   - 然后用 JavaScript + `docx` 包生成文档。
   - 如果需要验证文件是否真有内容，可检查：
     - 文件大小不是异常小空壳
     - `unzip -p 文件.docx word/document.xml | head`

6. **默认输出**
   - 先给用户一个 Word 文件。
   - 如用户还要改，再继续调整语气、长度、标题或内容详略。

## 重要约束

- 不要只靠猜测写周报，先找素材。
- 不要把口述笔记原样糊进去，要整理和归纳。
- 不要生成空白或只有标题的 Word。
- 如果文档生成链路失败，要明确说明失败点，并换路径处理。

## 本机经验（重要）

### 安装 skill / 资料查找
- 外部 skill 先审查来源与内容，再安装。
- 可以多路径查找：
  1. ClawHub / SkillHub
  2. awesome-openclaw-skills 这类索引仓库
  3. awesome-claude-skills / 官方 GitHub 仓库
- 对未知 skill，先看 `SKILL.md` 和相关文件，确认没有越权、偷数据、乱联网、摸密钥的行为。

### 本次 docx 处理的有效方案
- 官方 docx skill 可作为方法参考。
- 安装 Node 包：`npm install -g docx`
- 若 `require('docx')` 报 `MODULE_NOT_FOUND`，不要误判安装失败，先设置全局模块路径：
  - `export NODE_PATH="$(npm root -g)"`
- 然后再运行 Node 脚本生成 `.docx`。
